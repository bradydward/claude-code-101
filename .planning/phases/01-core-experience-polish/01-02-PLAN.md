  <name>Task 1: Audit Modules 1-8 for Completeness and Clarity</name>
  <files>curriculum.md</files>
  <action>
  Read curriculum.md and audit Modules 1-8. For each module, verify and fix:

  **Completeness Checks:**
  - Every lesson has numbered tasks (Task 1, Task 2, etc.)
  - No empty lessons or "TBD" placeholders
  - Module completion reward line exists (XP, Aura, stat, badge)
  - Stat Tag declared in module header

  **Clarity Improvements (rewrite where needed):**
  - Each task is ONE specific action (not multiple things in one task)
  - Commands shown in backtick code formatting
  - Expected output described ("you'll see..." or "it shows...")
  - Beginner-friendly language (no unexplained jargon)

  **WHY Explanations (add where missing):**
  Every task or group of related tasks must explain WHY:
  - Module 1: WHY terminal matters (it's how you talk to your computer directly)
  - Module 2: WHY API key (connects Claude Code to Claude's brain)
  - Module 3: WHY specific prompts (better input = better output)
  - Module 4: WHY different models (speed vs power tradeoff)
  - Module 5: WHY prompt patterns (templates save time, consistency)
  - Module 6: WHY plan mode (preview before action, like sketching before painting)
  - Module 7: WHY file types/paths (everything is files, knowing types = knowing tools)
  - Module 8: WHY git (save points for code, time travel for projects)

  Format WHY as natural language within the task, NOT as a separate labeled section. Example:
  - BAD: "Task 1: Type pwd. WHY: To see your location."
  - GOOD: "Task 1: Type `pwd` and press Enter (pwd = 'print working directory' = 'where am I?' - knowing your location is the first rule of the terminal)"

  **Single-Conversation Compatibility:**
  - Remove any "in your practice terminal" language if found
  - Tasks should just say "Type X" (works in any context)
  - For Claude Code tasks: "In Claude Code, type..." or "Ask Claude..."
  - For terminal tasks: "Type..." (student is already in terminal with Claude)
  </action>

<task type="auto">
  <name>Task 3: Verify WHY Coverage Across All 15 Modules (Comprehensive Check)</name>
  <files>curriculum.md</files>
  <action>
  After completing the audits in Tasks 1 and 2, run a comprehensive verification that WHY context exists across ALL 15 modules. Do NOT rely on spot-checking 2 modules.

  **Verification Method:**
  For each module (1-15), count the number of tasks that contain explanatory phrases. Explanatory phrases include:
  - Parenthetical explanations: `(` followed by explanation text `)` within a task
  - WHY indicators: "because", "so that", "this means", "this lets you", "the reason", "which means"
  - Analogy indicators: "like a", "think of it as", "similar to", "imagine"
  - Purpose phrases: "= '", "stands for", "short for"

  **Process:**
  1. For each module, count total tasks and tasks-with-WHY-context
  2. Calculate percentage: tasks_with_WHY / total_tasks * 100
  3. If any module has less than 50% WHY coverage, go back and add WHY explanations to those specific tasks

  **Expected Results (minimum thresholds):**
  - Modules 1-3 (beginner): 80%+ WHY coverage (everything is new, explain everything)
  - Modules 4-8 (intermediate): 60%+ WHY coverage (building on known concepts)
  - Modules 9-15 (advanced): 50%+ WHY coverage (student has more context, less hand-holding)

  **If a module fails threshold:**
  - Identify tasks missing WHY context
  - Add natural-language explanations (parenthetical, "because...", analogies)
  - Re-verify after additions

  **Output a verification table in the SUMMARY (not in curriculum.md):**
  | Module | Total Tasks | Tasks with WHY | Coverage % | Status |
  </action>
  <verify>
  Use grep to verify explanatory phrase density per module section:
  - For each module section, count lines matching: `\(.*\)|because|so that|this means|this lets|like a|think of|similar to|stands for|short for|which means|the reason`
  - Divide by total task lines in that module
  - ALL modules must show >= 50% coverage
  - Modules 1-3 must show >= 80% coverage
  - If any module fails, the task is NOT complete until fixed
  </verify>
  <done>
  All 15 modules verified to have WHY explanations meeting or exceeding their threshold (80% for Modules 1-3, 60% for 4-8, 50% for 9-15). Verification table produced showing exact counts per module. No module relies purely on "do this" instructions without explaining why.
  </done>
